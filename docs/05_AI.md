# 05 — AI Design

> **Status: design only.** The AI features are not implemented yet (scheduled
> for **Sprint 6** — AI Teacher/Tutor — and **Sprint 7** — recommendation). This
> document records the intended design. As of Sprint 3 the *storage target* and
> *personalization signals* the AI will use already exist: the `lessons`,
> `exercises`, and `submissions` tables, and `student_profiles.skill_level`.

## Components

| Component | Role |
|-----------|------|
| **AI Teacher** | Generates/explains lesson content for a topic at the learner's level. |
| **AI Tutor** | Reviews learner code, gives hints and feedback, answers questions. |

## Model

- Default to the latest, most capable Claude models.
- Suggested IDs (subject to change): `claude-opus-4-8` for high-quality teaching
  content, `claude-sonnet-4-6` for lower-latency tutoring/feedback.
- All model access goes through a single provider interface so models can be
  swapped via configuration.

## Boundary / Interface (planned)

AI lives behind an application-layer port so the rest of the system never calls
a provider SDK directly:

```python
# app/application/ports/ai_provider.py  (planned — not in Sprint 0)
class AIProvider(Protocol):
    async def teach(self, request: TeachRequest) -> TeachResponse: ...
    async def tutor(self, request: TutorRequest) -> TutorResponse: ...
```

A concrete `ClaudeAIProvider` will live in `infrastructure/ai/`.

## Data Flow (planned)

```text
Frontend ──POST /ai/teacher──▶ api ──▶ TeachUseCase ──▶ AIProvider (Claude)
                                          │
                                          └─ persists lesson/interaction in DB
```

## Content Generation — landing AI output in the existing schema

A core question: the courses, lessons, and exercises are ultimately meant to be
**AI-generated and personalized to the learner**, yet today they are authored
manually (admin CRUD + seed). These are not in tension — **AI generation does not
replace the schema, it populates it.** The manual CRUD built in Sprints 2–3 is
the foundation: it provides the storage target, the serving APIs, and the
progress tracking that AI generation drives.

Three pieces are already in place; Sprint 6 wires them together:

| Piece | Where it lives today |
|-------|----------------------|
| **Personalization signal** (what the learner knows) | `student_profiles.skill_level` (Sprint 1); `submissions` + grading results (Sprint 3–4); progress events (Sprint 7) |
| **Generation port** | `AIProvider.teach()` (this doc) → `ClaudeAIProvider` |
| **Storage target** | the `lessons` and `exercises` tables (Sprints 2–3) — the same rows the public `GET /courses`, `/lessons/{id}`, `/exercises/{id}` endpoints already serve |

### Data flow (planned, Sprint 6)

```text
learner level + history
  (student_profiles.skill_level, submissions, progress)
        │
        ▼
GenerateContentUseCase (application)
        │  builds a generation request: topic, target level, prior performance
        ▼
AIProvider.teach(request)  ──▶  Claude  ──▶  structured output
        │                                     (lesson markdown, or exercise:
        │                                      prompt + starter_code + test_spec)
        ▼
ContentService / repositories  ──▶  INSERT into lessons / exercises
        │
        ▼
served unchanged by  GET /courses/{slug} · /lessons/{id} · /exercises/{id}
graded unchanged by  the Sprint 4 Judge0 path (exercise.test_spec)
```

The generator writes through the **same `ContentService` and repositories** the
admin endpoints use — it is just another author. Nothing downstream (serving,
the editor, submissions, grading) needs to change.

### Mapping generated output to entities

| Generated artifact | Lands in |
|--------------------|----------|
| Lesson explanation | `lessons.content` (markdown) |
| Exercise statement | `exercises.prompt` |
| Editor starter code | `exercises.starter_code` |
| Hidden test cases | `exercises.test_spec` (JSONB) |
| Target language | `exercises.language` |

### Generation strategies

1. **Generate-and-persist** — store generated lessons/exercises as rows. Cheap to
   re-serve, cacheable, stable. Best for the course backbone.
2. **On-the-fly** — generate per request, not stored. Most personalized, but
   costlier and less reproducible. Best for "patch my weak spot" practice.
3. **Hybrid (recommended)** — a content pool (seeded + AI pre-generated) forms the
   backbone; per-learner gaps are filled with on-the-fly generation. The current
   architecture supports all three because generation sits behind `AIProvider`
   and writes through the existing repositories.

### Quality gates for generated content

- **Self-verification:** a generated exercise must include a `test_spec`; run the
  reference solution through the Sprint 4 Judge0 path before publishing, so we
  never ship an exercise whose own tests don't pass.
- **Provenance & review:** track whether a row was human- or AI-authored so admins
  can review/override AI drafts. (Planned: a `source` / `generated_by` column on
  `lessons`/`exercises`, added in the Sprint 6 migration — not present yet.)
- **Prompt-injection safety:** never feed raw learner input into generation
  prompts unescaped (see below).

### Personalization inputs (Sprint 7)

The recommendation engine selects/sequences content (and requests generation for
gaps) from: `skill_level`, per-topic mastery derived from `submissions` verdicts
and quiz attempts, and recency/streak from progress events. This is where
"based on the user's knowledge" is actually computed.

## Safety & Cost Controls (planned)

- Server-side prompt construction; never trust raw client prompts.
- Per-user rate limiting and token budgets.
- Streaming responses for tutoring UX.
- Prompt-injection mitigation on user-supplied code/questions.
- Logging of token usage for observability and billing.

## Out of Scope for Sprint 0

- Prompt templates, retrieval, evaluation harness, caching strategy.
- These are specified when Sprint 4 begins.
