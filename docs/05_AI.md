# 05 — AI Design

> **Status: design only.** No AI features are implemented in Sprint 0. This
> document records the intended design so later sprints can build against stable
> interfaces.

## Components

| Component | Role |
|-----------|------|
| **AI Teacher** | Generates/explains lesson content for a topic at the learner's level. |
| **AI Tutor** | Reviews learner code, gives hints and feedback, answers questions. |

## Model

- Default to the latest, most capable Claude models.
- Suggested IDs (subject to change): `claude-opus-4-8` for high-quality teaching
  content, `claude-sonnet-4-6` for lower-latency tutoring/feedback.
- All model access goes through a single provider interface so models can be
  swapped via configuration.

## Boundary / Interface (planned)

AI lives behind an application-layer port so the rest of the system never calls
a provider SDK directly:

```python
# app/application/ports/ai_provider.py  (planned — not in Sprint 0)
class AIProvider(Protocol):
    async def teach(self, request: TeachRequest) -> TeachResponse: ...
    async def tutor(self, request: TutorRequest) -> TutorResponse: ...
```

A concrete `ClaudeAIProvider` will live in `infrastructure/ai/`.

## Data Flow (planned)

```text
Frontend ──POST /ai/teacher──▶ api ──▶ TeachUseCase ──▶ AIProvider (Claude)
                                          │
                                          └─ persists lesson/interaction in DB
```

## Safety & Cost Controls (planned)

- Server-side prompt construction; never trust raw client prompts.
- Per-user rate limiting and token budgets.
- Streaming responses for tutoring UX.
- Prompt-injection mitigation on user-supplied code/questions.
- Logging of token usage for observability and billing.

## Out of Scope for Sprint 0

- Prompt templates, retrieval, evaluation harness, caching strategy.
- These are specified when Sprint 4 begins.
