## Model

- **Provider: Google Gemini**, via the **free tier** (Google AI Studio) ‚Äî this is
  a hackathon with no budget, so cost control is a hard constraint.
- Default: `gemini-2.5-flash-lite` for tutoring/feedback and generation ‚Äî its
  free-tier request quota is much higher than `gemini-2.5-flash` (whose free RPD
  is very low) and it's plenty for these structured tasks. Override
  `GEMINI_TEACHER_MODEL` to `gemini-2.5-pro`/`gemini-2.5-flash` for richer
  long-form content if you have the quota/budget. All model IDs are config-driven.
- Access uses the Google Gen AI SDK (`google-genai`) with an API key from
  [Google AI Studio](https://aistudio.google.com) (`GEMINI_API_KEY`).
- All model access goes through a single provider interface so the model ‚Äî or the
  provider ‚Äî can be swapped via configuration.

## Content Generation ‚Äî landing AI output in the existing schema

A core question: the courses, lessons, and exercises are ultimately meant to be
**AI-generated and personalized to the learner**, yet today they are authored
manually (admin CRUD + seed). These are not in tension ‚Äî **AI generation does not
replace the schema, it populates it.** The manual CRUD built in Sprints 2‚Äì3 is
the foundation: it provides the storage target, the serving APIs, and the
progress tracking that AI generation drives.

Three pieces are already in place; Sprint 6 wires them together:

| Piece | Where it lives today |
|-------|----------------------|
| **Personalization signal** (what the learner knows) | `student_profiles.skill_level` (Sprint 1); `submissions` + grading results (Sprint 3‚Äì4); progress events (Sprint 7) |
| **Generation port** | `AIProvider.teach()` (this doc) ‚Üí `GeminiAIProvider` |
| **Storage target** | the `lessons` and `exercises` tables (Sprints 2‚Äì3) ‚Äî the same rows the public `GET /courses`, `/lessons/{id}`, `/exercises/{id}` endpoints already serve |

admin endpoints use ‚Äî it is just another author. Nothing downstream (serving,
the editor, submissions, grading) needs to change.

### Generation strategies

1. **Generate-and-persist** ‚Äî store generated lessons/exercises as rows. Cheap to
   re-serve, cacheable, stable. Best for the course backbone.
2. **On-the-fly** ‚Äî generate per request, not stored. Most personalized, but
   costlier and less reproducible. Best for "patch my weak spot" practice.
3. **Hybrid (recommended)** ‚Äî a content pool (seeded + AI pre-generated) forms the
   backbone; per-learner gaps are filled with on-the-fly generation. The current
   architecture supports all three because generation sits behind `AIProvider`
   and writes through the existing repositories.

### Quality gates for generated content

- **Self-verification:** a generated exercise must include a `test_spec`; run the
  reference solution through the Sprint 4 Judge0 path before publishing, so we
  never ship an exercise whose own tests don't pass.
- **Provenance & review:** a `source` column (`human` | `ai`) on
  `lessons`/`exercises` (migration `0006_content_source`) marks AI-authored rows
  so admins can review/override AI drafts.
- **Prompt-injection safety:** never feed raw learner input into generation
  prompts unescaped (see below).

### Personalization inputs (Sprint 7)

The recommendation engine selects/sequences content (and requests generation for
gaps) from: `skill_level`, per-topic mastery derived from `submissions` verdicts
and quiz attempts, and recency/streak from progress events. This is where
"based on the user's knowledge" is actually computed.

## Safety & Cost Controls (planned)

- Server-side prompt construction; never trust raw client prompts.
- **Free-tier budget:** the Gemini free tier has per-minute/per-day request
  limits ‚Äî apply per-user rate limiting, cache/persist generated content
  (generate-and-persist) to avoid re-spending quota, and handle `429` quota
  errors gracefully (retry/backoff or a friendly "try again later").
- Streaming responses for tutoring UX.
- Prompt-injection mitigation on user-supplied code/questions.
- Logging of request/token usage for observability.

## Curriculum generation (Sprint 11)

`AIProvider.generate_syllabus` + `generate_lesson_pack` extend the port.
`CurriculumService` turns a track + level into a full course: it asks for a
syllabus, then for each topic generates a lesson pack (Markdown content +
exercises + a quiz) and writes it through the content/exercise/quiz repositories
with `source="ai"`. Generated exercises are self-verified (lenient ‚Äî see
Sprint 10) via Judge0. The heavy work runs in a background job
(`generation_jobs`) with progress polling; the manual content-authoring flow is
retired in favor of generation.

## Continuous learning (Sprint 12)

`CurriculumService.extend_course(focus, count)` appends more content to an
existing course via one primitive ‚Äî `generate_lesson_batch`, now with an optional
`focus`. "Learn more" continues the syllabus (no focus); a chat request passes a
short **transcript of the recent conversation** as the focus, so the model
resolves vague follow-ups ("I need more") against the most recent topic instead
of drifting to generic content. The learner chooses how many lessons to add
(`count`, bounded to `CURRICULUM_EXTEND_MAX`). New lessons get the next
`order_index` and `source="ai"`. `CourseChatService` persists the exchange
(`course_chat_messages`), calls `extend_course`, and replies with a summary of
what it added. Ownership (course must belong to the learner's track) and the
per-user AI budget are enforced. The "Learn more" hint appears once completion ‚â•
`CURRICULUM_EXTEND_THRESHOLD`.

## Plan-aware AI limits (Sprint 13)

AI usage is bounded per plan by `EntitlementService` (config-driven): the AI Tutor
and content generation/extend each have a rolling-24h daily cap counted from the
`ai_interactions` log (`kind="tutor"` / `"generate"`), free vs paid. Over-limit
raises `UpgradeRequiredError` ‚Üí HTTP `402`. This is separate from the free-tier
per-minute burst guard in `AIUsageGuard` (which stays a `429`). Admin-hidden
lessons (`review_status="hidden"`) are excluded from all learner-facing serving.

## Placement generation (Sprint 10)

`AIProvider.generate_placement` extends the port with a placement assessment
(multiple-choice + two coding tasks). `PlacementService` builds the items,
**self-verifies** each coding task's reference solution against its tests (Judge0)
before storing, strips answer keys on serve, grades a submission (coding weighted
higher), and maps the weighted score to a level on the track/profile ‚Äî the signal
Sprint 11 uses to pitch generated curricula.
